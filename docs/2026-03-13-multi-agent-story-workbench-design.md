# AI-Writing-Studio — Multi-Agent Story Workbench Design v0.3

## 1. 한 줄 정의
Markdown 기반 세계관 문서를 소스 오브 트루스로 삼고, 멀티에이전트가 캐릭터·사건·배경 맥락을 읽어 플롯과 장면 후보를 생성·검증하며, 인간 작가가 채팅형 제안 패널과 집필 에디터를 오가며 소설을 써나가는 Electron 기반 소설 워크벤치.

## 2. 왜 지금 이걸 하는가
기존 AI 글쓰기 도구는 대체로 다음 중 하나에 치우친다.

- 단순 채팅형 생성기: 세계관 누적과 구조적 추적이 약함
- 일반 문서 에디터: AI가 문맥을 깊게 읽고 움직이기 어려움
- 위키/노트 앱: 설정 정리는 좋지만 이야기 전개 시뮬레이션은 약함

이 프로젝트의 목적은 이 셋을 분리하지 않고 하나의 작업대 안에 통합하는 것이다.

핵심 문제의식은 다음과 같다.

1. 세계관과 캐릭터 정보가 Markdown 문서로 쌓여 있어야 사람이 직접 다룰 수 있다.
2. AI는 단일 응답기가 아니라 역할 분리된 에이전트로 협업해야 더 일관된 결과를 낼 수 있다.
3. 작가는 AI가 써준 결과를 본문에 강제로 반영당하는 대신, 채팅형 제안 인터페이스에서 검토하고 선택해야 한다.
4. 플롯 설계와 실제 집필은 분리되지만, 같은 작업대 안에서 자연스럽게 이어져야 한다.

## 3. 핵심 사용자 / 사용 상황
### 3.1 1차 사용자
- 본인 중심의 개인용 창작 도구 사용자
- 장편/연재/세계관 중심 서사를 쓰는 작가
- 캐릭터 관계, 설정, 사건 축적을 오래 유지하고 싶은 사용자

### 3.2 대표 사용 상황
1. 작가가 캐릭터, 장소, 사건, 규칙 문서를 Markdown으로 정리한다.
2. 먼저 플롯/사건 후보를 생성해본다.
3. 마음에 들지 않으면 플롯을 재생성하거나 수정한다.
4. 마음에 드는 플롯을 바탕으로 챕터를 집필한다.
5. 집필 중 대화나 장면 진행이 막히면 해당 챕터 문맥 기반으로 대화/반응 제안을 요청한다.
6. 작가는 채팅형 패널에서 추천을 보고 필요한 부분만 본문에 반영한다.

## 4. 제품 정체성
이 앱은 단순한 AI 에디터가 아니라 다음의 조합으로 정의된다.

- 멀티에이전트 스토리 엔진
- 세계관 문서 기반 지식 작업대
- 작가용 집필 에디터

즉, 정체성의 중심은 **에이전트 협업과 세계관 기반 추론**이며, 에디터는 그 결과를 집필로 연결하는 작업 공간이다.

## 5. 설계 원칙
1. **Markdown이 소스 오브 트루스다.**
   - 데이터베이스보다 사람이 읽고 편집 가능한 문서가 우선이다.
2. **구조화는 최소하지만 명확해야 한다.**
   - 본문은 자유롭게 두되, frontmatter/link/tag는 규약화한다.
3. **AI는 역할을 분리한다.**
   - 하나의 거대 프롬프트보다, 오케스트레이터 + 전문 에이전트 조합을 우선한다.
4. **작가가 항상 최종 결정권자다.**
   - 본문 자동 덮어쓰기는 하지 않고, 채팅형 제안과 선택 기반 반영을 중심으로 설계한다.
5. **근거를 보여준다.**
   - 결과가 어떤 문서와 관계에서 나왔는지 추적 가능해야 한다.
6. **확장은 그래프/RAG 방향으로 열어둔다.**
   - 초기에는 과설계를 피하되, 나중 확장을 막지 않는다.
7. **문서 스키마는 진화 가능해야 한다.**
   - 초기 폴더 구조와 필드는 고정 규칙이 아니라 확장 가능한 작업 규약으로 다룬다.
8. **집필 UX와 지식 관리 UX는 분리해서 최적화한다.**
   - 세계관 문서는 구조 편집 중심, 챕터는 리치 텍스트 집필 중심으로 설계한다.

## 6. 시스템 아키텍처 개요
추천 구조는 **Electron UI + 로컬 스토리 엔진 분리형**이다.

### 6.1 UI 레이어 (Electron)
책임:
- 문서 탐색
- 그래프/타임라인 시각화
- 집필 에디터
- 채팅형 에이전트 패널
- 에이전트 실행 요청 및 결과 표시

### 6.2 스토리 엔진 레이어 (로컬 서비스 또는 앱 내부 분리 모듈)
책임:
- Markdown 파싱
- frontmatter / 링크 / 태그 인덱싱
- 관련 문서 검색 및 컨텍스트 선택
- 멀티에이전트 실행 오케스트레이션
- 설정 충돌/타임라인 검증
- 세션 로그 저장
- 인증/공급자 어댑터 추상화

이 분리는 장기적으로 다음 확장에 유리하다.

- 모델 교체
- RAG 도입
- 그래프 쿼리 강화
- 배치 시뮬레이션
- 테스트 범위 분리

## 7. 기술 스택 결정 (초기안)
현재까지 합의된 초기 기술 스택은 다음과 같다.

- **앱 런타임**: Electron
- **언어**: TypeScript
- **프론트엔드**: React
- **빌드 툴**: Vite
- **상태 관리**: Zustand
- **스타일링**: Tailwind CSS
- **구조 문서 편집기**: CodeMirror 6
- **챕터 리치 텍스트 편집기**: Lexical
- **그래프 시각화**: React Flow
- **타임라인 시각화**: vis-timeline 계열 우선 검토
- **테스트**: Vitest + React Testing Library
- **Markdown 파싱**: gray-matter
- **스토리지**: 로컬 파일 시스템 기반

선정 이유는 다음과 같다.

1. React + Electron은 데스크톱 작업대 UI 구현 사례가 많고 자료가 풍부하다.
2. CodeMirror는 구조적 문서와 raw Markdown 편집에 적합하다.
3. Lexical은 무료/오픈소스 중심으로 리치 텍스트 집필 경험을 구성하기 좋다.
4. React Flow는 그래프 표현을 빠르게 붙이기 좋고, 작업대 UI와 통합성이 높다.
5. vis-timeline 계열은 초기 타임라인 가시화를 빠르게 확보하는 데 유리하다.

## 8. 데이터 모델 / 문서 구조
### 8.1 저장 형식
기본 저장 전략은 **혼합형**이다.

- 세계관 문서(`characters`, `events`, `locations`, `factions`, `rules` 등)는 Markdown(`.md`)을 사용한다.
- 챕터 문서는 리치 텍스트 편집 경험을 위해 **Lexical JSON 원본 저장**을 우선 채택한다.

즉, 세계관과 구조 지식은 Markdown이 소스 오브 트루스이고, 실제 집필 산출물인 챕터는 리치 에디터 포맷을 원본으로 관리한다.

예상 디렉터리 구조(초기안):

- `characters/`
- `events/`
- `locations/`
- `factions/`
- `rules/`
- `chapters/`
- `sessions/`

> 이 구조는 시작점이며, 작업하면서 `arcs/`, `notes/`, `relations/` 같은 타입으로 확장될 수 있다.

### 8.2 문서 구조
각 문서는 다음 3요소를 가진다.

1. **frontmatter**
2. **본문**
3. **링크 구조**

### 8.3 공통 frontmatter 필드 제안
```yaml
id: character-kira
type: character
title: 키라
tags: [주인공, 귀족, 마법사]
links:
  - event-first-duel
  - location-northern-castle
status: active
```

### 8.4 타입별 확장 필드 예시
#### character
- goals
- traits
- speech_style
- relationships
- perceptions
- awareness
- secrets

#### event
- participants
- causes
- consequences
- time_index
- status

#### location
- region
- atmosphere
- rules
- related_factions

### 8.5 관계 정보 설계 (초기안)
초기 버전에서는 별도 `relations/` 문서 타입보다, **캐릭터 문서 내부 구조**로 관계를 저장하는 방식을 우선 채택한다.

이때 관계는 최소 두 층으로 나눈다.

1. **objective relationship**
   - 객관적으로 존재하는 관계/사실
2. **subjective perception**
   - 해당 캐릭터가 믿고 있는 관계/오해/추측

필요 시 **awareness** 필드로 존재 인지 여부나 비밀 인지 여부를 저장한다.

예시:
```yaml
relationships:
  - target: character-b
    relation: mentor
    mutual: false
    notes: B는 A를 제자로 인정하지 않음

perceptions:
  - target: character-b
    belief: B는 나를 인정하고 있다
    truth_status: false

awareness:
  - target: character-c
    knows_existence: false
```

이 구조를 통해 다음 같은 비대칭 관계를 표현할 수 있다.
- A는 B를 스승이라고 믿음
- 실제로 B는 A를 제자로 인정하지 않음
- A는 C의 존재를 모름

복잡도가 높아지면 이후 `relations/` 문서 타입으로 승격 가능하다.

### 8.6 구조 전략
선택된 전략은 **혼합형**이다.

- 기본은 문서 단위 해석
- frontmatter / 링크 / 태그는 구조적으로 읽음
- 본문은 자유 서술 허용
- 관계와 인식 차이는 frontmatter로 기계 판독 가능하게 유지
- 챕터는 리치 텍스트 편집 상태를 JSON 원본으로 보존

이 구조는 Obsidian 스타일 작업과 호환성이 높고, 집필 UX 요구와 미래의 그래프/RAG 확장에도 유리하다.

## 9. 에이전트 설계
### 9.1 오케스트레이터
역할:
- 사용자 요청 해석
- 관련 문서/노드 수집
- 필요한 에이전트 선택
- 결과 통합 및 우선순위 정리
- 최종 출력 포맷 구성

### 9.2 캐릭터 에이전트
역할:
- 특정 인물의 말투, 동기, 관계, 인식 차이를 중심으로 반응 생성
- “이 인물이라면 어떻게 말하고 행동할까?”에 집중

중요 원칙:
- 서사 전체 통제보다 인물 일관성을 우선한다.

### 9.3 플롯 에이전트
역할:
- 다음 사건 후보 제안
- 장면 긴장도/갈등 구조 조정
- 전체 서사 흐름 관점에서 장면 기능 판단

### 9.4 세계관 검증 에이전트
역할:
- 설정 충돌 탐지
- 시간축 모순 탐지
- 인물 행동 불일치 점검
- 규칙 위반 및 개연성 경고
- 관계/인지 구조의 불일치 포착

### 9.5 검증 프로세스 원칙
검증은 선택 기능이 아니라 **필수 파이프라인**으로 둔다.

특히 문서 스키마, 에이전트 역할 정의, 프롬프트/행동 규칙은 이후 superpowers 기반 검토 루프를 돌릴 수 있게 설계한다.

### 9.6 실행 파이프라인
초기 기본 흐름은 다음과 같다.

1. 작가가 플롯 생성 요청
2. 플롯 에이전트가 사건/전개 후보 생성
3. 작가가 플롯을 채택, 수정, 재생성
4. 채택된 플롯을 기반으로 챕터 집필 시작
5. 집필 중 특정 장면이나 대화가 막히면 캐릭터 에이전트 호출
6. 검증 에이전트가 설정/시간/관계 충돌을 점검
7. 오케스트레이터가 결과를 통합해 채팅형 패널에 제시

즉, 핵심 루프는 **플롯 우선 → 집필 → 막히면 대화 추천 → 검증** 흐름이다.

## 10. UI / UX 설계
제품 형태는 **통합 작업대(workbench / cockpit)** 로 정의한다.

### 10.1 기본 레이아웃
- **좌측 패널**: 문서 트리, 태그 탐색, 그래프, 타임라인
- **중앙 패널**: 실제 집필 에디터
- **우측/하단 패널**: 채팅형 에이전트 패널, 경고, 추천, 근거 문서

### 10.2 채팅형 에이전트 패널
사용자에게는 기본적으로 **하나의 통합 어시스턴트 채팅**처럼 보이되, 필요하면 다음 관점으로 펼쳐볼 수 있다.

- 플롯 제안 보기
- 캐릭터 반응 보기
- 검증 경고 보기

즉, 외형은 통합형이되 내부 구조는 멀티에이전트인 혼합형(C) 구조를 채택한다.

### 10.3 편집기 전략
편집기는 문서 타입에 따라 분리한다.

- **세계관/설정 문서**: CodeMirror 기반 구조 문서 편집
- **챕터 문서**: Lexical 기반 리치 텍스트 집필 편집

즉, Markdown 소스 관리와 웹소설형 집필 UX를 하나의 도구 안에서 서로 다른 편집기로 최적화한다.

### 10.4 핵심 작업 흐름
1. 세계관 문서 작성 또는 수정
2. 플롯 생성 또는 수정
3. 플롯 확정 후 챕터 집필
4. 집필 중 막히는 장면/대화에 대해 채팅형 추천 요청
5. 결과 검토
6. 작가가 필요한 부분만 수동 반영
7. 검증 경고 확인 후 수정

### 10.5 UX 원칙
- 본문 자동 덮어쓰기 금지
- 결과는 채팅형으로 제안
- 생성 결과마다 근거 문서 링크 제공
- 경고는 숨기지 않고 명시적으로 표기
- 작가의 본문은 독립 자산으로 보호

## 11. 시각화 설계
그래프와 타임라인은 둘 중 하나만 선택하는 보조 기능이 아니라, 초기부터 함께 고려할 핵심 축이다.

### 11.1 그래프 뷰
목적:
- 인물 / 사건 / 장소 / 세력 간 연결을 시각적으로 탐색

활용:
- 특정 사건과 연결된 캐릭터 파악
- 관계 밀도 높은 노드 확인
- 누락된 연결 발견
- 인식 차이와 비대칭 관계 확인

### 11.2 타임라인 뷰
목적:
- 사건의 순서, 원인/결과, 진행 상태 추적

활용:
- 연표 점검
- 회상/분기/병렬 사건 정리
- 챕터와 사건 정렬
- 설정 모순 사전 발견

### 11.3 근거 하이라이트
생성 결과가 참조한 문서나 링크를 UI에 표시하여, 작가가 결과의 출처를 빠르게 검증할 수 있게 한다.

## 12. 모델 / 인증 관련 현실 제약과 방향
초기 논의 결과, 다음 전제를 둔다.

- 개인용 도구가 우선이다.
- 일반적인 OpenAI API key 직접 입력 방식은 선호하지 않는다.
- 특정 공급자 종속 설계는 피하고 싶다.
- 하지만 GPT 사용이 가능한 OAuth/세션 기반 흐름은 중요한 탐색 대상이다.

따라서 설계상으로는 다음 방향을 채택한다.

1. AI 호출 계층은 **공급자 어댑터 추상화**로 설계한다.
2. UX는 API key 입력을 기본 가정으로 두지 않는다.
3. GPT 사용이 가능한 OAuth/세션 기반 흐름은 **별도 리서치/검증 트랙**으로 다룬다.
4. 공식 지원 범위, 유지 가능성, 개인용 사용성은 별도 검증 후 구현 방향을 확정한다.

즉, 문서 차원에서는 이 요구를 반영하되, 아직 검증되지 않은 인증 흐름을 사실처럼 확정하지 않는다.

## 13. 장기 확장 방향
1. **RAG / 검색 최적화**
   - 링크 기반 탐색에서 벡터/하이브리드 검색으로 확장
2. **그래프 강화**
   - 문서 링크를 내부 지식 그래프로 투영
3. **시뮬레이션 로그 분석**
   - 어떤 에이전트가 어떤 제안을 냈는지 재사용
4. **문서 스키마 가이드 강화**
   - 문서 템플릿, 필수 필드 검사, lint 도입
5. **배치 시나리오 생성**
   - 여러 장면 후보를 한 번에 생성·비교
6. **관계 모델 승격**
   - 복잡도가 증가하면 `relations/` 문서 타입 분리
7. **편집기 저장 포맷 확장**
   - 필요 시 Lexical JSON ↔ export 포맷(HTML/Markdown/epub 등) 변환 계층 추가

## 14. 비목표 (초기 범위에서 제외)
초기 설계 단계에서 당장 핵심이 아닌 것:

- 공개용 SaaS 다중 사용자 계정 시스템
- 강한 서버 의존형 협업 기능
- 복잡한 결제/구독 구조
- 완전 자동 집필 파이프라인
- 과도하게 엄격한 DB 중심 스키마 강제

## 15. 구현 우선순위 제안
향후 계획 수립 시 우선순위는 다음 순서를 추천한다.

1. Markdown 문서 규약 확정
2. 캐릭터 관계/인지 구조 스키마 초안
3. 문서 파서 + 링크/메타 인덱서
4. 플롯 우선 실행 루프 정의
5. 오케스트레이터 초안
6. 캐릭터 / 플롯 / 검증 에이전트 인터페이스
7. 채팅형 에이전트 패널 포함 통합 작업대 UI 껍데기
8. 그래프 / 타임라인 기초 시각화
9. 인증/공급자 리서치 트랙
10. 검색/RAG 확장

## 16. 설계 결정 요약
- 플랫폼: Electron
- 프론트엔드: React + TypeScript + Vite
- 상태관리: Zustand
- 스타일링: Tailwind CSS
- 구조 문서 편집기: CodeMirror 6
- 챕터 편집기: Lexical
- 그래프: React Flow
- 타임라인: vis-timeline 계열 우선 검토
- 제품 정체성: 멀티에이전트 스토리 엔진 + 작가용 에디터
- 구조 방식: 문서 + frontmatter/link/tag 혼합형
- 저장 전략: 세계관은 Markdown, 챕터는 Lexical JSON 원본 저장
- 관계 정보는 초기엔 캐릭터 문서 내부 구조로 관리
- 코어 에이전트: 오케스트레이터 / 캐릭터 / 플롯 / 검증
- 핵심 루프: 플롯 우선 → 집필 → 막히면 대화 추천 → 검증
- UI 형태: 문서/집필/채팅형 에이전트 패널 통합 작업대
- 그래프와 타임라인을 모두 초기 설계 축으로 반영
- 아키텍처 추천: UI와 스토리 엔진 분리
