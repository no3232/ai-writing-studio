# AI-Writing-Studio — OpenClaw-Backed Story Workbench Design v0.4

## 1. 한 줄 정의
OpenClaw 호스트가 프로젝트 원본 저장소와 AI 오케스트레이션을 담당하고, Electron 앱은 원격 OpenClaw에 연결해 문서를 편집하고 플롯/장면/검증을 요청하는 소설 집필 워크벤치다.

## 2. 왜 지금 이걸 하는가
기존 AI 글쓰기 도구는 대체로 다음 중 하나에 치우친다.

- 단순 채팅형 생성기: 세계관 누적과 구조적 추적이 약함
- 일반 문서 에디터: AI가 문맥을 깊게 읽고 움직이기 어려움
- 위키/노트 앱: 설정 정리는 좋지만 이야기 전개 시뮬레이션은 약함

이 프로젝트의 목적은 이 셋을 분리하지 않고 하나의 작업대 안에 통합하는 것이다.

핵심 문제의식은 다음과 같다.

1. 세계관과 캐릭터 정보는 구조화 가능한 문서 저장소로 장기 축적되어야 한다.
2. AI는 단일 호출기가 아니라 문서 저장소 전체를 읽고 오케스트레이션할 수 있어야 한다.
3. 작가는 AI가 생성한 결과를 채택/수정/폐기하며 집필해야 한다.
4. 앱은 어떤 PC에서도 실행되되, AI 백엔드는 OpenClaw가 설치된 호스트에서 담당할 수 있어야 한다.

## 3. 핵심 사용자 / 사용 상황
### 3.1 1차 사용자
- 본인 중심의 개인용 창작 도구 사용자
- 장편/연재/세계관 중심 서사를 쓰는 작가
- 캐릭터 관계, 설정, 사건 축적을 오래 유지하고 싶은 사용자
- 여러 기기에서 같은 프로젝트와 AI 백엔드에 붙고 싶은 사용자

### 3.2 대표 사용 상황
1. OpenClaw 호스트 PC에 프로젝트 저장소 원본이 있다.
2. 사용자는 데스크톱 또는 노트북의 Electron 앱에서 해당 호스트에 접속한다.
3. 앱은 프로젝트를 열고 문서 목록, 챕터, 그래프, 타임라인을 표시한다.
4. 사용자는 플롯 생성 / 장면 제안 / 검증 요청을 보낸다.
5. OpenClaw는 프로젝트 저장소를 직접 읽고 관련 문서를 모은 뒤 오케스트레이션을 수행한다.
6. 결과는 스트리밍 또는 응답 형태로 앱에 돌아오고, 작가는 채팅형 패널을 통해 선택적으로 반영한다.

## 4. 제품 정체성
이 앱은 단순한 AI 에디터가 아니라 다음의 조합으로 정의된다.

- OpenClaw 기반 AI 오케스트레이션 클라이언트
- 세계관 문서 기반 지식 작업대
- 작가용 집필 에디터

즉, 앱의 정체성은 **OpenClaw 위에 올라가는 창작 전용 클라이언트**에 가깝고, OpenClaw는 프로젝트 저장소와 AI 문맥을 소유하는 백엔드 역할을 맡는다.

## 5. 설계 원칙
1. **프로젝트 원본 저장소는 OpenClaw 호스트가 소유한다.**
2. **앱은 thin client에 가깝게 유지한다.**
3. **OpenClaw 세부 구현은 브리지 뒤에 숨긴다.**
4. **문서 CRUD와 AI 오케스트레이션을 분리한다.**
5. **작가가 항상 최종 결정권자다.**
6. **장기적으로 RAG/그래프/검증 강화에 유리한 구조를 택한다.**
7. **브리지는 점진적으로 분리 가능해야 한다.**

## 6. 시스템 아키텍처 개요
추천 구조는 **Electron 클라이언트 + 앱 내부 OpenClaw 브리지 + 원격 OpenClaw 호스트**다.

### 6.1 클라이언트 레이어 (Electron)
책임:
- 프로젝트 탐색
- 문서 편집
- 챕터 집필
- 그래프/타임라인 시각화
- 채팅형 AI 제안 패널
- 사용자 요청/반영 UI

### 6.2 브리지 레이어 (`openclaw-bridge`)
초기에는 앱 내부 모듈로 두고, 필요하면 별도 companion process로 분리한다.

책임:
- OpenClaw 호스트 연결 관리
- 호스트 URL / 접속 토큰 관리
- 문서 CRUD 요청 래핑
- AI 요청 래핑
- 스트리밍 응답 처리
- 재시도 / 타임아웃 / 에러 처리

### 6.3 서버 레이어 (OpenClaw host)
책임:
- 프로젝트 저장소 원본 보관
- 문서 스캔/인덱싱
- 관계 파악
- 플롯/장면/검증 오케스트레이션
- 장기적으로 RAG/검색/그래프 계산 담당

## 7. 통신 구조
앱은 OpenClaw를 직접 tool 수준으로 호출하지 않고, **브리지가 도메인 API로 변환**한다.

### 7.1 1차 연결 방식
- 호스트 URL
- 접속 토큰

### 7.2 브리지가 노출할 API 범주
#### 프로젝트/문서 API
- `project.list`
- `project.open`
- `document.list`
- `document.get`
- `document.create`
- `document.update`

#### AI API
- `plot.generate`
- `scene.suggest`
- `validation.run`
- `graph.snapshot`
- `timeline.snapshot`

### 7.3 스트리밍
브리지는 request/response뿐 아니라 streaming abstraction도 제공해야 한다.

## 8. 기술 스택 결정 (초기안)
- **앱 런타임**: Electron
- **언어**: TypeScript
- **프론트엔드**: React
- **빌드 툴**: Vite
- **상태 관리**: Zustand
- **스타일링**: Tailwind CSS
- **구조 문서 편집기**: CodeMirror 6
- **챕터 리치 텍스트 편집기**: Lexical
- **그래프 시각화**: React Flow
- **타임라인 시각화**: vis-timeline 계열 우선 검토
- **테스트**: Vitest + React Testing Library
- **Markdown 파싱**: gray-matter
- **브리지 구현**: 앱 내부 TypeScript 모듈 (`openclaw-bridge`)로 시작

## 9. 프로젝트 원본 / 문서 구조
### 9.1 원본 저장 위치
프로젝트 원본 저장소는 OpenClaw 호스트 PC에 존재한다.

### 9.2 저장 형식
- 세계관 문서(`characters`, `events`, `locations`, `factions`, `rules`)는 Markdown(`.md`) 사용
- 챕터 문서는 **Lexical JSON 원본 저장**

### 9.3 문서 디렉터리 구조 (초기안)
- `characters/`
- `events/`
- `locations/`
- `factions/`
- `rules/`
- `chapters/`
- `sessions/`

### 9.4 CRUD 전략
앱은 원격 저장소를 직접 파일 시스템처럼 다루지 않고, 반드시 브리지를 통해 조작한다.

1차 문서 작업 범위:
- 문서 목록
- 문서 열기
- 문서 생성
- 문서 수정

## 10. OpenClaw 중심 AI 설계
OpenClaw 호스트가 프로젝트 저장소를 직접 읽고 오케스트레이션을 담당한다.

코어 에이전트:
- 오케스트레이터
- 캐릭터 에이전트
- 플롯 에이전트
- 세계관 검증 에이전트

실행 흐름:
1. 사용자가 앱에서 요청
2. 브리지가 도메인 요청으로 변환
3. OpenClaw가 프로젝트 저장소를 직접 읽고 관련 문서 수집
4. 오케스트레이션 수행
5. 결과를 스트리밍 또는 응답으로 앱에 반환
6. 앱은 이를 채팅형 패널, 그래프, 타임라인, 집필 에디터에 반영

## 11. UI / UX 설계
- 좌측: 프로젝트/문서 트리, 그래프, 타임라인
- 중앙: 집필 에디터
- 우측/하단: 채팅형 AI 패널, 경고, 추천, 근거 문서

편집기 전략:
- 세계관/설정 문서: CodeMirror
- 챕터 문서: Lexical

## 12. OpenClaw 연결 / 인증 방향
- 앱은 인터넷 원격 OpenClaw 호스트에 연결될 수 있어야 한다.
- 초기 연결 방식은 **호스트 URL + 접속 토큰**
- Pairing UX나 고급 인증은 후속 단계
- 브리지는 초기엔 앱 내부 모듈, 필요하면 별도 companion service로 승격

## 13. 장기 확장 방향
- RAG / 검색 최적화
- 그래프 강화
- 시뮬레이션 로그 분석
- 관계 모델 승격
- 브리지 분리
- OpenClaw 호스트 다중 연결

## 14. 비목표 (초기 범위에서 제외)
- 공개용 SaaS 다중 사용자 계정 시스템
- 강한 서버 의존형 협업 기능
- 복잡한 결제/구독 구조
- 완전 자동 집필 파이프라인
- 초기 단계에서의 고급 pairing UX

## 15. 설계 결정 요약
- 제품 정체성: OpenClaw 기반 멀티에이전트 창작 클라이언트
- 원본 저장소: OpenClaw 호스트 측 프로젝트 저장소
- 저장 전략: 세계관은 Markdown, 챕터는 Lexical JSON 원본 저장
- 코어 에이전트: 오케스트레이터 / 캐릭터 / 플롯 / 검증
- UI 형태: 문서/집필/채팅형 AI 패널 통합 작업대
- 통신 구조: 앱 내부 브리지 + 원격 OpenClaw 호스트
- 연결 범위: 인터넷 원격 OpenClaw까지 고려
- 브리지 전략: 초기엔 앱 내부, 나중에 분리 가능(C)
